# 定时器锁相代码依赖关系说明

## 1. 文档范围

本文档对应当前 `TRIPLE_ADC_14M_MODE = 0` 的主运行路径，也就是：

- `COMP1 + TIM3` 负责采集输入信号边沿
- `COMP2 + TIM2` 负责采集 DAC 反馈边沿
- `TIM6 + DAC + DMA + DDS` 负责连续输出正弦
- 软件在输入捕获中断中完成频率跟踪和相位校正

文档不把三路高速 ADC 交织采样模式作为主线，只在需要时补充说明。

## 2. 一句话总览

这份代码的核心结构可以概括为：

`输入边沿时间戳` 和 `DAC 反馈边沿时间戳` 进入 `UpdateComparatorLockOnInputEdge()`，该函数计算当前频率和相位误差，再去更新 `dac_phase_step` 与 `dac_phase_accumulator`；之后 `FillDacBuffer()` 在 DAC DMA 回调中读取这两个状态，生成新的正弦输出。

也就是说：

- 锁相控制发生在 `TIM3` 输入捕获中断里
- 波形输出发生在 `TIM6/DAC DMA` 回调里
- 两条链路通过共享的 DDS 状态量耦合在一起

## 3. 运行时主链路

### 3.1 初始化链路

主函数中的调用顺序如下：

1. `SystemClock_Config()`
2. `MX_GPIO_Init()`
3. `MX_DMA_Init()`
4. `MX_DAC1_Init()`
5. `MX_TIM2_Init()`
6. `MX_TIM3_Init()`
7. `MX_TIM6_Init()`
8. `MX_UART5_Init()`
9. `MX_COMP1_Init()`
10. `MX_COMP2_Init()`
11. `InitDacGenerator()`
12. `HAL_TIM_IC_Start_IT(TIM2, CH4)`
13. `HAL_TIM_IC_Start_IT(TIM3, CH1)`
14. `StartDacTimerDma()`
15. `SendStartupBannerUart5()`

依赖关系说明：

- `MX_COMP1_Init()` 和 `MX_TIM3_Init()` 必须先完成，输入边沿才能被送到锁相控制函数
- `MX_COMP2_Init()` 和 `MX_TIM2_Init()` 必须先完成，DAC 反馈边沿才能被正确采样
- `InitDacGenerator()` 必须在 `StartDacTimerDma()` 之前完成，否则 DMA 启动后没有有效波形数据
- `StartDacTimerDma()` 之后，DAC 输出链路进入持续运行状态

### 3.2 运行期链路

运行期的主要关系如下：

```text
外部输入 -> COMP1 -> TIM3输入捕获 -> HAL_TIM_IC_CaptureCallback()
                                     -> UpdateComparatorLockOnInputEdge()
                                     -> 更新 dac_phase_step / dac_phase_accumulator

DAC输出 -> COMP2 -> TIM2输入捕获 -> HAL_TIM_IC_CaptureCallback()
                                   -> 更新 comp_feedback_capture_ticks

TIM6更新事件 -> DAC DMA半满/全满回调 -> FillDacBuffer()
                                      -> 读取 dac_phase_step / dac_phase_accumulator
                                      -> 生成后续正弦数据
```

## 4. 核心函数依赖表

| 函数 | 谁调用它 | 它依赖什么 | 它会更新什么 | 作用 |
| --- | --- | --- | --- | --- |
| `InitDacGenerator()` | `main()` | `FrequencyToPhaseStep()`、`FillDacBuffer()`、DAC 常量宏 | `dac_sine_lut`、`dac_output_frequency_hz`、`dac_target_frequency_hz`、`dac_phase_step`、`dac_phase_accumulator` | 生成 DDS 查找表并建立 DAC 初始状态 |
| `StartDacTimerDma()` | `main()` | `hdac1`、`htim6` | 启动硬件，不直接写锁相状态 | 启动 DAC DMA 和 TIM6 采样时钟 |
| `FillDacBuffer()` | `InitDacGenerator()`、`HAL_DAC_ConvHalfCpltCallbackCh1()`、`HAL_DAC_ConvCpltCallbackCh1()` | `dac_phase_step`、`dac_phase_accumulator`、`dac_sine_lut` | `dac_dma_buffer`、`dac_phase_accumulator` | 依据当前 DDS 状态生成输出波形 |
| `FrequencyToPhaseStep()` | `InitDacGenerator()`、`UpdateComparatorLockOnInputEdge()` | 目标频率和 DAC 采样率 | 无全局写入 | 把频率换算成 DDS 相位步进 |
| `WrapPhaseErrorTicks()` | `UpdateComparatorLockOnInputEdge()` | 原始 tick 差值、周期 tick | 无全局写入 | 把时间差折返到最小等效相位误差范围 |

| `ReportLockStatusUart5()` | `main()` 主循环 | 各类 `comp_*`、`dac_*` 状态 | `uart_report_count`、`uart_last_report_tick` | 周期性打印调试状态 |
| `HAL_TIM_IC_CaptureCallback()` | HAL 中断框架 | TIM2/TIM3 捕获寄存器 | `comp_feedback_capture_ticks`，或间接触发大量锁相状态更新 | 分发两路输入捕获事件 |
| `HAL_DAC_ConvHalfCpltCallbackCh1()` | HAL DMA 中断框架 | `FillDacBuffer()` | `dac_dma_half_count` | DAC 前半缓冲重装 |
| `HAL_DAC_ConvCpltCallbackCh1()` | HAL DMA 中断框架 | `FillDacBuffer()` | `dac_dma_full_count` | DAC 后半缓冲重装 |

## 5. 关键共享状态的依赖关系

### 5.1 `dac_phase_step`

- 写入者：
  - `InitDacGenerator()`
  - `UpdateComparatorLockOnInputEdge()`
- 读取者：
  - `FillDacBuffer()`

含义：

- 它决定 DDS 每输出一个采样点时，相位前进多少
- 频率越高，`dac_phase_step` 越大

依赖关系结论：

- 锁相控制函数通过修改 `dac_phase_step` 改变后续输出频率
- DMA 填充函数通过读取 `dac_phase_step` 把这种频率变化变成真实波形

### 5.2 `dac_phase_accumulator`

- 写入者：
  - `InitDacGenerator()`
  - `FillDacBuffer()`
  - `UpdateComparatorLockOnInputEdge()`
- 读取者：
  - `FillDacBuffer()`

含义：

- 它表示 DDS 当前“跑到哪一个相位位置”
- `FillDacBuffer()` 会持续推进它
- `UpdateComparatorLockOnInputEdge()` 会在必要时直接拨动它，快速校正相位

依赖关系结论：

- `FillDacBuffer()` 负责自然推进相位
- `UpdateComparatorLockOnInputEdge()` 负责有控制地“纠偏”
- 两边都可能访问它，所以代码里用 `__disable_irq()` 保护读写一致性

### 5.3 `comp_feedback_capture_ticks`

- 写入者：
  - `HAL_TIM_IC_CaptureCallback()` 中的 `TIM2/CH4` 分支
- 读取者：
  - `UpdateComparatorLockOnInputEdge()`

含义：

- 它保存最近一次 DAC 反馈边沿的时间戳
- 本身不触发控制，只作为输入信号锁相时的参考边沿

依赖关系结论：

- `TIM2` 分支负责提供“输出侧观测值”
- `TIM3` 分支在控制计算时消费这个观测值

### 5.4 `comp_nominal_frequency_hz` / `comp_candidate_frequency_hz`

- 写入者：
  - `UpdateComparatorLockOnInputEdge()`
- 读取者：
  - `UpdateComparatorLockOnInputEdge()`
  - `ReportLockStatusUart5()`

含义：

- `comp_input_frequency_hz` 是每次边沿估出来的瞬时频率
- `comp_nominal_frequency_hz` 是量化后、确认稳定的目标频点
- `comp_candidate_frequency_hz` 是等待确认切换的新候选频点

依赖关系结论：

- 锁相不是直接跟随瞬时频率
- 而是先经过“量化 + 候选确认”后，才更新 DAC 的目标频率

## 6. 锁相控制函数的内部依赖顺序

`UpdateComparatorLockOnInputEdge()` 自身也有严格的内部依赖顺序：

1. 读取当前输入边沿时间戳
2. 与上一次输入边沿做差，计算输入周期
3. 对周期做范围检查和短窗口平均
4. 由周期计算输入频率
5. 把输入频率量化到 nominal 频点，并做候选确认
6. 用 `TIM2` 保存的 DAC 反馈时间戳计算当前边沿差
7. 用 `WrapPhaseErrorTicks()` 折返边沿差
8. 如果还没有建立参考相位，就记录 `comp_phase_reference_ticks`
9. 如果参考相位已建立，则计算本次相位误差
10. 用相位误差积分出 `comp_trim_hz`
11. 用 `FrequencyToPhaseStep()` 计算新的 `dac_phase_step`
12. 直接微调 `dac_phase_accumulator`

这几个步骤不能随便交换，原因如下：

- 没有输入周期，就没有频率
- 没有 nominal 频率，就没有稳定目标
- 没有 DAC 反馈边沿，就无法计算相位差
- 没有相位误差，就不能更新频率微调和相位补偿

## 7. 中断与主循环的职责边界

### 7.1 中断里做什么

- `HAL_TIM_IC_CaptureCallback()`
  - 处理输入边沿与反馈边沿
  - 执行锁相控制
- `HAL_DAC_ConvHalfCpltCallbackCh1()`
  - 重装前半段 DAC DMA 缓冲
- `HAL_DAC_ConvCpltCallbackCh1()`
  - 重装后半段 DAC DMA 缓冲

### 7.2 主循环里做什么

- `ReportLockStatusUart5()`
  - 定时输出当前状态

结论：

- 当前设计是“中断驱动控制，主循环只做观测”
- 所以即使主循环很简单，锁相仍然能够持续工作

## 8. 如果你继续改代码，建议优先关注的依赖点

### 8.1 改锁相参数时

优先关注：

- `COMP_TRIM_KI_HZ`
- `COMP_TRIM_MAX_HZ`
- `DAC_LOCK_PHASE_ALPHA`
- `COMP_FREQ_SWITCH_CONFIRM_COUNT`

因为这些参数会直接改变 `UpdateComparatorLockOnInputEdge()` 对 `dac_phase_step` 和 `dac_phase_accumulator` 的更新力度。

### 8.2 改输出链路时

优先关注：

- `FillDacBuffer()`
- `FrequencyToPhaseStep()`
- `DAC_SAMPLE_RATE_HZ`
- `DAC_LUT_SIZE`

因为这些内容决定“锁相结果最终如何转成模拟波形”。

### 8.3 改边沿采集时

优先关注：

- `MX_COMP1_Init()`
- `MX_COMP2_Init()`
- `MX_TIM2_Init()`
- `MX_TIM3_Init()`
- `HAL_TIM_IC_CaptureCallback()`

因为锁相算法的全部输入都来自这几处。

## 9. 对照代码时的推荐阅读顺序

建议按下面顺序阅读源码：

1. `main()`
2. `InitDacGenerator()`
3. `StartDacTimerDma()`
4. `HAL_DAC_ConvHalfCpltCallbackCh1()` / `HAL_DAC_ConvCpltCallbackCh1()`
5. `FillDacBuffer()`
6. `HAL_TIM_IC_CaptureCallback()`
7. `UpdateComparatorLockOnInputEdge()`
8. `WrapPhaseErrorTicks()`
9. `FrequencyToPhaseStep()`
10. `ReportLockStatusUart5()`

按这个顺序看，最容易把“初始化 -> 中断 -> 控制 -> 输出 -> 调试”整个闭环串起来。
